# craft-sync

Детерминированный детектор изменений для Craft через **Multi-Document REST API**.
Вынесен из агента: тяжёлый deep-fetch всех деревьев блоков и сравнение дат делает
бинарник, агент получает на stdout **только изменившиеся страницы** (id + путь).
Гигантский JSON в контекст агента не попадает.

## Зачем

Старый алгоритм актуализации памяти проверял `lastModifiedAt` только корневого
блока документа (`GET /blocks ... maxDepth=0`). В Craft дата **не всплывает вверх
через границу страницы**, поэтому правка во вложенной заметке (например «Цель
проекта» в проекте «Старый домен CRM», изменена 04.06 при корне 01.06) терялась.

`craft-sync` скачивает всё дерево (`maxDepth=-1`, `fetchMetadata=true`), сворачивает
дату до **гранулярности страницы** (для каждого блока-`page` берётся максимум
`lastModifiedAt` его самого и всех потомков, кроме вложенных страниц — те считаются
отдельными единицами), и сравнивает с постраничным снэпшотом.

## Использование

```bash
craft-sync \
  --base "$CRAFT_API_BASE" \           # https://connect.craft.do/links/XXXX/api/v1
  --docs DOC1,DOC2,DOC3 \              # корневые id сфер (получает агент через MCP folders/documents)
  --exclude INFRA_DOC1,INFRA_DOC2 \   # доки «Память для агента» (роутер/снэпшот/SKILL)
  --snapshot /path/snapshot.json \    # читается для диффа
  --update-snapshot \                 # перезаписать снэпшот свежими датами
  --incremental \                     # пропустить deep-fetch доков, чья дата не выросла (экономит бюджет)
  --tree                              # человекочитаемое дерево (без флага — JSON)
```

- `--docs` пуст → бинарник сам зовёт `GET /documents` и берёт все неудалённые.
  Но REST API **не знает о папках**, поэтому ограничить обход папкой «Сферы» и
  отсечь подпапку «Память для агента» можно только передав явные id (агент берёт
  их дёшево через MCP `folders list`/`documents list`) или через `--exclude`.
- `--since RFC3339` — запасной режим без снэпшота: страница считается новой, только
  если новее этой даты.
- Битые/призрачные id (`Block not found`) не валят обход — попадают в `errors`.

### Инкрементальный режим (`--incremental`) — экономия бюджета

REST API считает **бюджет блоков**: после большого deep-fetch следующий запрос
может вернуть `HTTP 429 "Block budget exceeded"`. Чтобы не качать всё дерево
каждый раз, `--incremental` сначала берёт дешёвый `GET /documents`
(только метаданные, бюджет не тратит) и сравнивает doc-level `lastModifiedAt`
с порогом. Порог берётся так: если есть снэпшот с картой `docs` — это `docs[<id>]`;
если снэпшота нет — это значение `--since`. Если дата **не выросла** над порогом —
документ не скачивается (`scanned` ↓, `skipped` ↑); при наличии снэпшота его
страницы переносятся как есть.

Это безопасно, потому что doc-level дата в листинге `/documents` **всплывает
через границы страниц** (в отличие от per-block метаданных в `/blocks`, на чём и
основан баг root-only). Проверено на «Старый домен Crm»: листинг отдаёт дату
вложенной правки (`2026-06-04T10:22:47Z`), а собственная дата корневого блока
(`maxDepth=0`) осталась на `2026-06-01T06:33:38Z` — на 3 дня старее.

Серверного фильтра по дате у connect-link нет (`?modifiedSince=` игнорируется),
поэтому отсев делается на клиенте по листингу. Связка `--since` + `--incremental`
**без снэпшота** — основной режим: одна сохранённая дата-cutoff отсекает все
неизменившиеся доки по дешёвому листингу, а deep-fetch получают только те, чья
дата позже cutoff. Без снэпшота и без `--since` прогон полный.

### Устойчивость к лимиту (`--retries`, `--rl-retries`)

- `--retries` (по умолч. 3) — повторы для сетевых ошибок и `5xx`, backoff 2/4/8с.
- `--rl-retries` (по умолч. 5) — отдельные повторы для `HTTP 429`, с более долгим
  backoff 5/10/20/40/60с (capped), чтобы окно бюджета успело пополниться. Раньше
  `429` не ретраился и ронял документ в `errors`; теперь обход сам пережидает.

### Формат вывода (JSON)

```json
{
  "changed": [
    {"id":"PG_GOAL","title":"Цель проекта #заметка","status":"changed",
     "lastModifiedAt":"2026-06-04T13:22:18+03:00","parentId":"DOC_CRM",
     "path":["Старый домен CRM 🤖"]}
  ],
  "unchangedCount": 18, "totalPages": 20, "scannedDocs": 21, "errors": []
}
```

### Формат снэпшота

```json
{"generatedAt":"...","version":7,
 "docs":{"<rootDocId>":"<doc-level lastModifiedAt из /documents>"},
 "pages":{"<pageId>":"<RFC3339 lastModifiedAt>"},
 "titles":{"<pageId>":"<title>"},
 "pageDoc":{"<pageId>":"<rootDocId>"}}
```

Постраничный, а не подокументный — в этом и есть фикс. `docs` и `pageDoc`
добавлены для `--incremental`: `docs` хранит doc-level дату для решения
«качать/пропустить», `pageDoc` — принадлежность страницы корневому доку, чтобы
перенести страницы пропущенного дока без повторного скачивания.

## Бэклинки: кто ссылается на блок (`--backlinks`)

Нативного backlinks-API у Craft нет: ни MCP, ни connect-link REST не отдают
входящие ссылки, а поисковый индекс покрывает только видимый текст — ID цели
живёт в атрибуте ссылки и не находится ничем, включая RE2-regex-поиск по ID
(проверено экспериментально: `search --regexp "<block-id>"` и
`GET /documents/search?regexps=<id>` дают 0 совпадений на документе, который
текстовый поиск видит). Зато **исходящие** ссылки полностью присутствуют в
markdown каждого блока как `[текст](block://<id>)` — по ним строится
инвертированный индекс.

```bash
craft-sync --backlinks <blockId> --links-file /path/links.json   # запрос одним вызовом
craft-sync --links-refresh --links-file /path/links.json         # только прогрев индекса (для рутины)
craft-sync --backlinks <id> --links-file ... --offline           # ответ из индекса без сети
```

- Инкрементальность та же, что у детектора изменений: дешёвый `GET /documents`,
  deep-fetch только доков с выросшей датой, остальные записи переносятся из
  `--links-file`. Замер на живом пространстве (39 доков, 1065 ссылок): первичная
  стройка ~49 с, повторный запрос ~0.8 с, `--offline` ~0.03 с.
- `--docs` сужает множество ПЕРЕсканируемых доков (записи остальных не
  выбрасываются), `--exclude` выкидывает док целиком. Сравнение ID —
  регистронезависимое.
- Ловятся обе формы ссылок: `[текст](block://id)` (с текстом) и
  `craftdocs://...blockId=` / голый `block://` (без текста). Удалённые доки и
  доки, пропавшие из листинга, выпадают из индекса при следующем рефреше.
- Ответ: JSON с `backlinks[]` (source-блок, его док, ближайшая страница,
  цепочка заголовков страниц, текст ссылки), `scannedDocs`/`skippedDocs`,
  `stale` при ответе без рефреша.

⚠️ Покрытие = scope connect-ссылки: ссылки из документов вне шаринга в индекс
не попадают (сам блок-цель вне scope тоже не проблема — важен доступ к
документам-ИСТОЧНИКАМ). Новые документы в шаринг сами не попадают. Для 100%
покрытия connect-ссылка должна видеть всё пространство. Вторая слепая зона —
ссылки, нарисованные внутри вайтбордов (их элементы отдаются отдельным
endpoint и в дерево `/blocks` не входят). Элементы коллекций индексируются.

### Тёплый индекс в репозитории

`craft-sync/links.json` закоммичен в репо как тёплый старт: свежая сессия
получает готовый индекс с клоном, и первый же `--backlinks`-вызов перекачивает
только изменившиеся с последнего коммита доки (секунды вместо ~50 с и полного
расхода block-бюджета). Обновляет его ежедневная облачная рутина:
`--links-refresh` + коммит `links.json` в main, только если файл изменился.
Протухший коммит индекса — не ошибка корректности: даты в нём всё равно
заставят перекачать всё изменившееся, протухание стоит только времени.

## ⚠️ Предусловия окружения

1. **Сеть.** Egress окружения ходит через allowlist-прокси; хост
   `connect.craft.do` должен быть разрешён в сетевой политике окружения
   (настройки Claude Code on the web,
   см. https://code.claude.com/docs/en/claude-code-on-the-web). В текущем
   окружении хост открыт — режим `--backlinks` проверен живыми вызовами.
2. **Эфемерность.** Контейнер пересоздаётся. `/usr/local/bin/craft-sync` не
   переживёт пересоздание — исходник в `/home/user/craft-sync/` нужно хранить в
   репозитории или пересобирать в SessionStart-хуке:
   `cd /home/user/craft-sync && go build -o /usr/local/bin/craft-sync .`
3. **Токен.** Connect-link-токен зашит в URL `--base`. Держать его в env
   `CRAFT_API_BASE` (переменные окружения настраиваются в окружении), а не в коде.
